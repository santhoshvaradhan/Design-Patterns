# Adapter Pattern - Notification Service

This document explains the Adapter Pattern and analyzes the implementation in this project (`Notification.Service.Adapter-Pattern`).

## Definition

Adapter Pattern lets you convert the interface of a class into another interface clients expect. It allows classes with incompatible interfaces to work together by wrapping an existing class with a new interface.

## When to use

- You have an existing class (or third-party library) whose interface does not match the one your code expects.
- You need to integrate a legacy system without changing its code.
- You want to reuse code that cannot be modified but must conform to your application's interface.

## Why use it

- Decouples client code from external/legacy implementations.
- Keeps a single, stable interface for clients while allowing adapters to evolve.
- Makes testing easier by enabling mockable target interfaces.

## Where to use (real-world scenarios)

- External libraries (e.g., an email sending library named `External.Email.Lib`).
- Third-party APIs with different request/response models (e.g., SMS gateways, push notification providers).
- Legacy systems with incompatible interfaces that you cannot change.

## Analysis of adapter code in this project

- `INotificationService.cs` (target interface): defines the methods the application expects for sending notifications. Clients depend on this interface.
- `EmailAdapter.cs` (adapter): implements `INotificationService` and internally uses the external email library to send emails. The adapter translates calls from the `INotificationService` shape into the shape required by the external library.
- `Program.cs` (composition/usage): shows how the adapter is instantiated and used by the application, keeping the client code independent of the external library.

This structure keeps the application code simple and decoupled:

- Client -> `INotificationService` -> `EmailAdapter` -> `External Email Library`

## Example cases

1) External library

   - Problem: `External.Email.Lib` exposes a method `Send(EmailMessage message)` but your app uses `INotificationService.SendNotification(NotificationRequest req)`.
   - Solution: Create `EmailAdapter` that maps `NotificationRequest` to `EmailMessage` and calls `External.Email.Lib`.

2) Third-party API

   - Problem: Multiple providers (SendGrid, Twilio) have different method signatures.
   - Solution: Create an adapter per provider implementing `INotificationService`. Register desired adapter in composition root.

3) Legacy system

   - Problem: Legacy notification system has blocking synchronous calls and old DTOs.
   - Solution: Adapter wraps legacy calls, performs mapping and optionally executes calls on a background thread or task to match modern async APIs.

## Minimal adapter example (concept)

This is the mapping idea that `EmailAdapter` implements in this project.

```csharp
// Target interface expected by application
public interface INotificationService
{
    Task SendNotification(NotificationRequest request);
}

// Adapter implementation
public class EmailAdapter : INotificationService
{
    private readonly ExternalEmailClient _client; // from external lib

    public EmailAdapter(ExternalEmailClient client)
    {
        _client = client;
    }

    public async Task SendNotification(NotificationRequest request)
    {
        var message = new EmailMessage
        {
            To = request.Recipient,
            Subject = request.Title,
            Body = request.Body
        };

        await _client.SendAsync(message);
    }
}
```

## Best practices

- Keep adapters thin: only transform data and delegate calls.
- Do not put business logic into adapters.
- Prefer constructor injection for adapter dependencies to make testing easier.
- Provide one adapter per external/legacy implementation to keep concerns isolated.
